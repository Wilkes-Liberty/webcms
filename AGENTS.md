# AGENTS.md — Wilkes Liberty Web CMS (Drupal)

This guide orients AI agents working in the `webcms/` repository — the headless Drupal 11 backend for wilkesliberty.com.

For the overall platform architecture, three-repo relationships, environments, and deployment, start with the root [../AGENTS.md](../AGENTS.md) and `infra/docs/ENVIRONMENT_OVERVIEW.md`.

## Project Overview

**Headless Drupal 11 CMS** serving content via JSON:API and GraphQL to a Next.js 15 frontend. The system is multilingual (English / Spanish / Russian) and uses **PostgreSQL 16** (not MySQL) in all environments.

**Key characteristics**:
- Configuration-driven: almost everything lives in `config/sync/` and is managed via Drush `cex`/`cim`.
- 9 content types with a consistent tab-based field group pattern.
- Heavy use of Paragraphs for flexible Landing Pages.
- Custom GraphQL schema extensions and JSON:API enhancements in `web/modules/custom/wl_api`.
- 8 active Composer patches (see `docs/PATCHES.md` and `composer.json`).
- Tight integration with Keycloak (OpenID Connect), Redis cache, Solr search, and the Next.js "Next" module for preview + on-demand revalidation.
- AI-assisted content tools via the `ai` module suite.

## Local Development (DDEV + PostgreSQL 16)

**Always use DDEV** for local work. It is configured to match production (PostgreSQL 16, PHP 8.3, nginx-fpm).

```bash
# One-time / daily
ddev start
ddev composer install          # after composer.lock changes
ddev drush cim -y              # after pulling config changes
ddev drush cr

ddev launch                    # opens https://api.wilkesliberty.dev
ddev launch /admin
ddev psql                      # PostgreSQL CLI (never mysql)
```

**Daily discipline (non-negotiable)**:
- After any change in the Drupal admin UI that affects configuration entities → immediately:
  ```bash
  ddev drush cex -y
  git add config/sync/
  git commit -m "Export configuration: <description of change>"
  ```
- After pulling someone else's config changes → `ddev drush cim -y && ddev drush cr`

See `docs/local-development.md` for settings hierarchy, Xdebug, cache disabling, and troubleshooting.

**Settings files** (in order of precedence):
1. `settings.php` (committed base — includes the OIDC `$config[]` override block, see below)
2. `settings.ddev.php` (auto-generated by DDEV)
3. `settings.local.php` (your overrides, gitignored; useful for disabling caches during dev)

## Local HTTPS (mkcert)

DDEV serves trusted HTTPS at both `https://api.ddev.site` and `https://api.wilkesliberty.dev` once mkcert's local CA is installed in the System keychain:

```bash
mkcert -install      # one-time, prompts for sudo to add CA to System keychain
ddev restart         # regenerates the project cert (api.ddev.site + api.wilkesliberty.dev in SAN)
```

If browsers show a TLS warning, the CA is missing from the System keychain. Run `mkcert -install` again. The project cert is at `.ddev/traefik/certs/api.crt` and includes both DDEV hostnames in its SAN list.

## Keycloak SSO Architecture (per-environment clients)

Each Drupal environment uses its own Keycloak client; the legacy shared `drupal` client was deleted on 2026-05-26.

| Environment | KC client | Redirect URI whitelist |
|---|---|---|
| Production | `drupal-prod` | `api.wilkesliberty.com`, `api.int.wilkesliberty.com` |
| Staging | `drupal-staging` | `api-stg.int.wilkesliberty.com` |
| Local DDEV | `drupal-local` | `api.ddev.site`, `api.wilkesliberty.dev` (http + https for both) |

**Runtime override**: `web/sites/default/settings.php` contains a `$config[]` block that reads `DRUPAL_OIDC_CLIENT_ID` and `DRUPAL_OIDC_CLIENT_SECRET` env vars and overrides `openid_connect.client.sign_in_with_keycloak` at request time. This means:
- The committed config_sync file holds a placeholder, not a real secret.
- Pulling production data into staging/local does NOT overwrite the per-env client wiring.
- A leaked staging or local secret cannot be used against prod KC.

**Where the secrets live**:
- Prod + staging: SOPS-encrypted in `infra/ansible/inventory/group_vars/sso_secrets.yml` (`kc_drupal_prod_client_secret`, `kc_drupal_staging_client_secret`); rendered into `~/nas_docker/.env` / `~/nas_docker_staging/.env` by Ansible; consumed by docker-compose env vars.
- Local DDEV: `.ddev/config.local.yaml` (gitignored) sets `web_environment` with `DRUPAL_OIDC_CLIENT_ID=drupal-local` + the secret.

**Source of truth for client definitions**: `infra/scripts/keycloak/setup-realm.sh` (`drupal-prod`, `drupal-staging`, `drupal-local` blocks).

## Public-API Lockdown

`https://api.wilkesliberty.com` (the public Drupal hostname) is locked down by the VPS Caddy to OIDC + files routes only — no admin, no GraphQL, no OAuth endpoints. Operators log in via the internal Tailscale URL `https://api.int.wilkesliberty.com`. Do not assume the public hostname can be used for admin or content authoring.

## Content Modeling — The Authoritative Sources

When working on content types, fields, displays, or editorial workflows, these documents (in `docs/`) are the source of truth:

| Document | Purpose | When to update |
|----------|---------|----------------|
| **FIELD_REFERENCE.md** | Canonical specification of every field (name, type, cardinality, required, allowed values, description) | After any field addition/change |
| CONTENT_TYPES_GUIDE.md | Purpose and editorial usage of the 9 content types | When workflows or purposes evolve |
| FIELD_FORMATS_GUIDE.md | Text formats, AI features, allowed HTML | When formats change |
| PARAGRAPHS.md | All paragraph bundles used on Landing Pages + component guidance | When adding new paragraph types |
| CONTENT.md | Editorial workflows, moderation, publishing | Process changes |

**Rule**: `FIELD_REFERENCE.md` wins on any conflict.

### The 9 Content Types

| Machine Name | Human Name | Primary Use |
|--------------|------------|-------------|
| article | Article | News, blog posts, announcements, press |
| basic_page | Basic Page | Static pages (About, Policies, Legal) |
| career | Career | Job postings |
| case_study | Case Study | Client success stories |
| event | Event | Webinars, conferences, appearances |
| landing_page | Landing Page | Marketing / campaign pages (Paragraphs-driven) |
| person | Person | Team bios, author profiles, leadership |
| resource | Resource | Downloadable gated content (eBooks, guides) |
| service | Service | Consulting / service offerings |

**Taxonomy vocabularies** (important for relationships and navigation):
Solutions, Technologies, Capabilities, Industries, Topics, Tech Stack, Personas, Compliance, and several others.

Content uses a consistent tab/field-group organization in the admin form: Content / Media / CTAs / SEO / Technical / Relationships / Classification / Layout / Editorial.

## Custom Modules (`web/modules/custom/`)

- **wl_api** — Primary place for custom REST/JSON:API helpers, GraphQL extensions, and any API surface the Next.js frontend depends on. Watch its CHANGELOG.
- **wl_language_switcher** — Custom language switching UI (the site is EN/ES/RU).
- **wl_taxo_nav** — Keeps taxonomy terms in sync with the main menu.
- **wl_text_formats** — Custom text format and filter configurations.
- **wl_scripts** — One-off Drush scripts and bulk operations.

When you add new API behavior that the UI consumes, document it in the PR and consider updating types/queries in the `ui/` repo.

## API Surface (what the Next.js frontend consumes)

- **JSON:API**: `/jsonapi` (standard Drupal) + custom endpoints under `wl_api`.
- **GraphQL**: `/graphql` (primary for the frontend; uses `graphql` + `graphql_compose` modules heavily).
- The Next.js app also uses the `next` module (drupal/next) for:
  - On-demand ISR revalidation (shared secret)
  - Draft/preview mode (shared secret)
  - OAuth2 consumer for authenticated preview requests

When changing the GraphQL schema (new fields, new types, new resolvers), the frontend queries in `ui/lib/queries/` will need corresponding updates.

## Working with Composer Patches

There are currently 8 active patches (see `composer.json` → `extra.patches` and the `patches/` directory).

Each patch has an explanation in `docs/PATCHES.md`. Before adding a new patch:
1. Search the Drupal issue queue for an existing issue.
2. Create one if needed with clear reproduction steps.
3. Document the root cause and the minimal diff in `docs/PATCHES.md`.
4. Add the patch entry to `composer.json`.

Never edit files inside `web/core/` or `vendor/` directly — they are overwritten on `composer install`.

## Cross-Repo Coordination

This repo does **not** live in isolation:

- Changes to content types, fields, or GraphQL schema almost always require corresponding work in `ui/`.
- When you export new config that adds fields the frontend queries, mention it explicitly in the PR and coordinate with frontend developers.
- Deployment of the Drupal image is driven by `infra/docker/drupal/Dockerfile.*` + Ansible. The build context reaches into this repo from `infra/`.
- Secrets for the OAuth2 consumer that Next.js uses live in `infra/ansible/inventory/group_vars/app_secrets.yml` (SOPS) and are injected at runtime. See the bootstrap ordering in the root AGENTS.md if you are doing a fresh install.

**When making API-affecting changes**, always:
1. Update `FIELD_REFERENCE.md` (or the relevant docs/).
2. Export and commit config.
3. Open a linked PR or issue in the `ui/` repo describing the new data shape.
4. Test against the UI dev server pointed at your DDEV instance.

## Common Commands & Gotchas

**Config hygiene**
```bash
ddev drush cex -y          # after admin changes
ddev drush cim -y          # after git pull of config
ddev drush config:status   # what differs from sync
ddev drush cr
```

**Database**
```bash
ddev psql
ddev export-db --file=backup-$(date +%Y%m%d).sql.gz
ddev import-db --src=backup.sql.gz
```

**Refreshing production data locally (or staging)**
- See `docs/local-development.md` → "Refreshing Production Data"
- **Best practice**: After `make refresh-staging`, run:
  ```bash
  ~/Repositories/infra/scripts/create-dev-snapshot.sh --from-staging
  ```
  Then on developer machines:
  ```bash
  ./scripts/refresh-env.sh --target local --fetch -y
  ```
  The script now strongly prefers these sanitized snapshots.
- Canonical full staging refresh: `make -C ../infra refresh-staging`

**Debugging**
```bash
ddev drush watchdog:show          # recent logs
ddev drush pm:list | grep -E 'graphql|next|jsonapi'
ddev xdebug on                    # then attach from IDE on port 9003
```

**Gotchas**
- This project uses **PostgreSQL 16**, not MySQL. Use `psql`, not `mysql`.
- `config/sync/` is the single source of truth for everything that can be exported. Do not rely on the database for long-lived configuration.
- OAuth2 consumer credentials for the Next.js frontend are created **after** the first Drupal install (see app_secrets.yml bootstrap in infra docs).
- During local dev you may need to disable render/page caches in `settings.local.php` (example in `docs/local-development.md`).
- When the GraphQL schema changes, the Next.js dev server may need a restart to pick up new types.
